GH Archive · 2026-07 · relational bot detection

What the graph found, and what the model didn't

Structural queries over GitHub event data surfaced two botnets and a market in purchased engagement. The relational model, given the same graph and a task built so that per-account features carry nothing, stayed at chance.

The headline

The graph works

Twelve hand-verified accounts expanded to 191 by joining on a single relational key. A second, unrelated network of 139 accounts fell out of the same query. Neither was reachable from any per-account feature.

The model did not

On a task where every entity row is identical — so nothing but the graph can separate the classes — RelativeDB scored 0.479 AUC. That is chance. The head trained cleanly (loss 0.707 → 0.516, 64 distinct scores across 66 accounts), so this is not overfitting; it simply did not find the structure a groupby finds instantly.

Botnet 1: 191 accounts from 12 seeds

Found by joining repos on their base name rather than owner/name

The verified accounts don't share repos — they each create their own copy of the same-named repo. Under owner/name those are 191 distinct nodes with one actor each, which is exactly why co-occurrence analysis found nothing and why repo_actors = 1.0 for every one of them.

Twelve were already verified by hand; the join pulled 179 more out of 336k events. The login style is consistent — Daemonflovibrate27, Goldenclewalk, ChordFlamingo — generated two-word handles.

This is a schema gap, not just a query trick

repos has no node for the repo name itself. A repo_name → base_name link would put an account one hop from a name shared with 190 others, making the ring reachable by traversal rather than by an offline join.

Botnet 2: 139 accounts, none previously known

The same query surfaced a second component with no overlap with the first: names are zapret-discord-youtube, discordzapret, zapret-discord-1.9.9c — a Russian DPI-bypass tool, mass-reuploaded across 139 accounts with the same generated-handle style (AlchemistIntensify, BurrowEmirBoil).

Two components in the same output were not botnets: 7,023 accounts joined through generic names (.github, dotfiles, portfolio), and 74 accounts sharing neetcode-submissions — a class assignment. Shared names alone are not proof; the name has to be distinctive.

Every purchased account has exactly one event

One star or one fork, nothing else, all day. That is the entire footprint, and it is why a per-account model cannot work: the row is empty. The only signal is that 36 such accounts converged on one repo.

Why it failed, as far as I can tell

The purchased accounts star real projects — TigerVNC/tigervnc, mattsse/chromiumoxide, SWE-agent/SWE-ReX — because that is the product: the customer wants a legitimate repo to look popular. So the neighbourhood is not distinctive. A bought star and a genuine one land on the same repo, and within a single day both accounts have one event.

The convergence signal is real but rare: 36 one-event accounts on grok-build, out of 2,046 accounts total. Too sparse to train on.

The co-visitor structure grows roughly twice as fast as the data, because a purchased batch spreads across days: a repo showing 3 one-event visitors on day one shows 20 across the week. That is the scaling argument, and it is the one thing here that favours the method — but it is an argument about the data, not a measurement of the model on it.

Labels

13 disguised bots, 32 infrastructure bots and 63 humans, all verified by hand. Humans were judged by reading their issue text — first person, a specific version, a complaint, a typo — not by counting anything, since counts are what raised the suspects in the first place.

Accounts whose issues are template output (Upptime monitors, Power Automate, AI assistants posting under a person's token) carry no label: the account is a person and the writing is not.

Honest verdict

The relational structure is real and findable. Two botnets, 330 accounts, from twelve seeds and one join key.

RT-J did not find it. Three attempts — repo-star signature, co-occurrence, purchased engagement — all landed at chance, including on a task built so that the entity row carries no information at all.

What is untested is the slope. Every RelativeDB number here came from a single training size — 44, 64, 98 rows. Flat at chance across 4× more labels on a 6.5× denser graph would be the kill signal; rising would mean these numbers are a small-sample reading. That measurement was not made.
